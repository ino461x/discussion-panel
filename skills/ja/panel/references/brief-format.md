# Brief Format — ブリーフ抽出ルール

パネリストを起動する前に、トピックを構造化ブリーフに要約する。
各カテゴリを個別に抽出する — パネリストはこのデータの異なるビューを受け取る。

## 基本フォーマット

```
トピック:          具体的な質問または決定事項 (1-2文)
ステークス:        この判断を誤った場合、何が起こるか？

技術的制約:
  - ハードな技術的制限、パフォーマンス要件、プラットフォーム制約
  - 具体的な数値: 行数、関数数、依存関係数など
  - 関連する実際の障害事例や既知のバグ

ビジネス的制約:
  - 納期、予算、チーム規模、コンプライアンス要件

ユーザー行動:
  - エンドユーザーがシステムとどのように実際にやり取りしているか

暗黙の前提:
  - 明確な根拠なしに真とされていること

ユーザーの主張:
  - ユーザーが表明した立場とその背後にある論理
  - これが自明に思えても含めること — Critic と Contrarian の主要な攻撃対象

過去の試行:
  - 含める: 「Xを試した結果、Yが観察された」(事実に基づく結果)
  - 除外: 「Xはダメだ / Xは機能しない」(パネリストに X 再検討を避けさせるバイアスになる)
```

## 動的カテゴリ化

**全カテゴリを必ず埋める必要はない。** トピックに関連するカテゴリだけ抽出する:

- 純粋な技術トピック (リファクタリング、バグ修正、パフォーマンス): 技術的制約とユーザー行動に集中、
  ビジネス的制約は該当時のみ
- プロダクト判断 (機能追加、UX方針): ユーザー行動とビジネス的制約を厚く
- アーキテクチャ選択: 技術的制約 + 過去の試行 + 暗黙の前提を重視
- 非技術トピック: 技術的制約カテゴリを省略

該当しないカテゴリは書かない。空欄を埋めるためだけに情報を水増ししない。

## トピックが曖昧な場合

事実/制約のない1行のトピックでは、抽象的で役に立たない分析になる。
続行前に1つの明確化質問を行う。例:

- 「どの時点での判断を迷っていますか？ (実装前 / 実装中 / レビュー後)」
- 「代替案として何を既に検討しましたか？」
- 「この判断で最も心配なことは何ですか？」

1つだけ聞く。複数聞くとユーザーの労力が増え、パネル起動より答える方が大変になる。

# Discussion — 多角的分析パネル

## このスキルが存在する理由

AIと1対1で作業していると、微妙だが危険なパターンが生まれる:

1. **確証バイアスの漂流** — AIがあなたのフレーミングに適応し、反論しなくなる
2. **コンテキストのトンネルビジョン** — 双方とも既に議論された内容しか見えなくなる
3. **場当たり的な修正** — 根本原因に対処せず、表面的なパッチが蓄積する
4. **疑われない前提** — 誰も再検証しないため、誤った仮定が生き残る

このスキルは、現在の会話の結論に何の利害関係も持たない新鮮なサブエージェントを
起動することで、このパターンを打破する。サブエージェントがトピックを独立して分析し、
その後あなたとメインエージェントが一緒にその発見をレビューする。

これが何であるかを正直に述べておく: **真に独立した専門家ではなく、同一の基盤モデルによる
構造化されたセルフレビューである。** 新鮮なコンテキストが会話の惰性を取り除き、
構造化された役割がメインの会話で見落とされた角度の検証を強制するため、機能する。

## 呼び出し方

/discussion [トピック]                標準 (パネリスト2名)
/discussion full [トピック]           フルパネル (パネリスト4名)
/discussion max [トピック]      MAXパネル (パネリスト5名)
/discussion [トピック] --ctx          コードベース探索を強制 (技術トピックではデフォルトON)
/discussion [トピック] --no-ctx       コードベース探索を明示的に無効化

`--ctx` デフォルト: 技術トピック (コード、アーキテクチャ、バグ) ではON、非技術トピックではOFF。
ユーザーは `--ctx` / `--no-ctx` で上書き可能。

モード指定なしで呼び出された場合、トピックの重要度を評価する:

#### 軽量トピック (アーキテクチャへの影響なし、数分で巻き戻し可能、単一ファイルスコープ)
質問不要。Standard + Sonnet をサイレントに実行する。

#### 中程度 / 重量級トピック
AskUserQuestion を使用して両方の質問をまとめて提示する。トピックの重要度に応じて
推奨オプションを設定する (中程度 → Full + Balanced、重量級 → Max + All Opus)。

{
  "questions": [
    {
      "question": "パネルの規模を選んでください。",
      "header": "Scale",
      "multiSelect": false,
      "options": [
        {
          "label": "Standard",
          "description": "2視点 (Critic, Realist)。軽い確認向き"
        },
        {
          "label": "Full (Recommended)",
          "description": "4視点 (+ Architect, Outsider)。設計判断に最適"
        },
        {
          "label": "Max",
          "description": "5視点 (+ Contrarian)。重大な意思決定向き"
        }
      ]
    },
    {
      "question": "パネリストのモデルを選んでください。",
      "header": "Model",
      "multiSelect": false,
      "options": [
        {
          "label": "All Sonnet",
          "description": "全員 Sonnet。高速・低コスト"
        },
        {
          "label": "Balanced (Recommended)",
          "description": "重要パネリストが Opus、他は Sonnet (コスパ最良)"
        },
        {
          "label": "All Opus",
          "description": "全員 Opus。最高精度だがコスト大"
        }
      ]
    }
  ]
}

トピックの重要度に応じて "(Recommended)" ラベルを調整する:
- **中程度**: Full + Balanced を推奨
- **重量級**: Max + All Opus を推奨

#### Balanced モデル割り当て

現在のモードに含まれるパネリストにのみ適用される (Standard は Critic + Realist のみ)。

| パネリスト | Balanced モードのモデル | 理由 |
|----------|-------------------|-----|
| **Critic** | Opus | 深い前提への挑戦にはより強力な推論が必要 |
| **Architect** | Opus | 構造的・体系的分析には深い洞察が有利 |
| **Contrarian** | Opus | 整合性のある反論の構築は最も難度が高いタスク |
| **Realist** | Sonnet | 実践的なトレードオフ評価は Sonnet で十分対応可能 |
| **Outsider** | Sonnet | 初心者目線のシンプルさチェックに Opus は不要 |

ユーザーがモードやモデルを明示的に指定した場合は、質問をスキップする。
ユーザーが「とりあえずやって」と言ったり、急いでいる様子であれば、推奨デフォルトをサイレントに使用する。

## パネリスト一覧

各パネリストには認知フレームワークと Starting Artifact がある。Starting Artifact は
所見を出す前に必ず完了しなければならない思考エクササイズである。これにより、
同一の基盤モデルからでも真の多様性が生まれる。

### Standard モード — パネリスト2名:

| 役割 | フォーカス | 認知フレームワーク | Starting Artifact |
|------|-------|---------------------|-------------------|
| **Critic** | 欠陥のある前提、見落としたリスク、代替フレーミング、何がうまくいかないか | プレモーテム + 5 Whys | まず3つの失敗シナリオを書く。それを分析の出発点とする。 |
| **Realist** | 実装コスト、保守負担、よりシンプルな代替案、「これは投資に見合うか？」 | 具体的な見積もりに基づくコスト対効果分析 | まず実装コストをフェーズ別の人日で見積もる。 |

### Full モード — パネリスト4名 (追加):

| 役割 | フォーカス | 認知フレームワーク | Starting Artifact |
|------|-------|---------------------|-------------------|
| **Architect** | 根本原因、体系的影響、長期的帰結、設計の一貫性、見落とされたシナジーと再利用機会 | 第一原理からの分解 | まず依存関係のチェーンを描く。意見を形成する前に、何が何に依存しているかをマッピングする。 |
| **Outsider** | 不要な複雑さ、分かりにくい命名、「なぜ単純に...しないのか？」という問い | 初心者の目 + 異分野アナロジー | ソフトウェア以外の分野で、同じ種類の問題を解決した事例を1つ挙げる。そこから出発する。 |

### Max モード — パネリスト5名 (追加):

| 役割 | フォーカス | 認知フレームワーク | Starting Artifact |
|------|-------|---------------------|-------------------|
| **Contrarian** | 現在のアプローチとは正反対の最も強力な主張を構築する — 単に欠陥を見つけるのではなく (それは Critic の仕事)、整合性のある代替方向を築く | スチールマン・インバージョン | 現在のアプローチが一度も提案されなかった世界を想像する。代わりに何が構築され、なぜそれが明らかに正しいとされたかを記述する。 |

Contrarian は最もコストが高いが、しばしば最も価値のあるパネリストである。
選ばれた道への批判だけでなく、選ばれなかった道を真剣に検討することを強制する。

## 実行フロー

### ステップ 1: コンテキスト抽出

パネリストを起動する前に、トピックを構造化されたブリーフに要約する。
ブリーフには4つのファクトカテゴリに加え、ユーザーの主張と過去の試行がある。
各カテゴリを個別に抽出する — パネリストはこのデータの異なるビューを受け取る。

トピック:          具体的な質問または決定事項 (1-2文)
ステークス:        この判断を誤った場合、何が起こるか？

技術的制約:
  - [ハードな技術的制限、パフォーマンス要件、プラットフォーム制約]
  - [具体的な数値: 行数、関数数、依存関係数など]
  - [関連する実際の障害事例や既知のバグ]

ビジネス的制約:
  - [納期、予算、チーム規模、コンプライアンス要件]

ユーザー行動:
  - [エンドユーザーがシステムとどのように実際にやり取りしているか、観察されたパターン]

暗黙の前提:
  - [明確な根拠なしに真とされていること]

ユーザーの主張:
  - [ユーザーが表明した立場とその背後にある論理]
  - これが自明に思えても含めること — Critic と Contrarian の主要な攻撃対象である。

過去の試行:
  - 含める: 「Xを試した結果、Yが観察された」(事実に基づく結果)
  - 除外: 「Xはダメだ / Xは機能しない」(パネリストがXの再検討を避けるようバイアスをかける結論)

**トピックが短すぎるまたは曖昧な場合**、続行前に1つの明確化質問を行う。
事実/制約のない1行のトピックでは、抽象的で役に立たない分析になる。

### ステップ 1.5: 情報配分

各パネリストはブリーフの異なるビューを受け取る。現在のモードに含まれるパネリスト
のビューのみ構成する (Standard = Critic + Realist のみ)。

| パネリスト | 受け取る情報 |
|----------|----------|
| **Critic** | 全カテゴリ + 暗黙の前提をトップに目立つように配置。追加: 「このブリーフが扱っていない質問を3つ挙げよ — 目立って欠落しているものは何か？」 |
| **Realist** | 全カテゴリ + 技術的制約とビジネス的制約を強調 |
| **Architect** | 全カテゴリ + 技術的制約 + 依存関係/構造情報を強調 |
| **Outsider** | トピックとステークスのみ — 制約なし、ユーザーの主張なし、過去の試行なし (意図的な白紙状態) |
| **Contrarian** | 全カテゴリ + ユーザーの主張をトップに目立つように配置 |

**Outsider の情報衛生**: Outsider のステークス記述から実装固有の用語 (関数名、
ファイルパス、ライブラリ名) を除去する。ステークスはビジネス/ユーザー影響の
言葉のみで記述する。非技術トピックの場合はステークス自体を省略し、
Outsider にはトピック文のみを渡す。

#### ステップ 1.5b: 動的 Artifact パラメータ化

パネリストを起動する前に、ステップ 1 のトピック固有のコンテキストを各役割の
Starting Artifact 指示 (上記パネリスト一覧テーブルより) に注入する。
関連する要素を追加する: Critic には攻撃すべき具体的な暗黙の前提、
Realist には見積もるべき具体的なアプローチ、Architect にはマッピングの
起点となる主要コンポーネント、Contrarian には反転すべきユーザーの主張を渡す。
Outsider には注入しない (白紙状態を維持)。

フォールバック: 複数の候補がある場合は最も具体的なものを優先する。デフォルト:
Critic → 最初の暗黙の前提、Realist → アプローチの最もコストの高いフェーズ、
Architect → 最も依存関係の多いコンポーネント、Contrarian → ユーザーの核心的主張。

### ステップ 2: パネリストの起動

全パネリストを単一メッセージ内で Agent ツールを使用して**並列に**起動する。
確定した構成に従い、各 Agent 呼び出しに `model` パラメータを設定する。

以下のパネリストプロンプトテンプレートを各パネリストに使用し、一覧とステップ 1.5 から
`[ROLE]`、`[FRAMEWORK]`、`[ARTIFACT_INSTRUCTION]`、`[BRIEF_VIEW]` を代入する。

#### パネリストプロンプトテンプレート

```
あなたはレビューパネルの [ROLE] である。このプロジェクトについて以下に提供される情報以外の
事前コンテキストは一切持たない — これはバイアスの継承を避けるための意図的な設計である。

## あなたの視点: [ROLE]
[一覧テーブルのフォーカス文]

## 認知フレームワーク: [FRAMEWORK]

## Starting Artifact (必須 — スキップ不可。所見はこのエクササイズから導出すること)
[ARTIFACT_INSTRUCTION]
続けて: ここで書いた内容から、自明でない含意を1つ特定する。

## あなたの情報ビュー
[ステップ 1.5 の BRIEF_VIEW — パネリスト固有のサブセット]

## 出力フォーマット
1. **Starting Artifact** — 上記エクササイズを完全に書き出す
2. **Reasoning chain** — 最も重要な発見を150-200語で展開する。
   Starting Artifact から導出されなければならない。「[具体的な要素を名指しする —
   例: 失敗シナリオ2、マッピングした隠れた依存関係、フェーズ3のコスト見積もり] から...」
   で始め、一般的なパターンマッチングではなくエクササイズに推論を固定する。
3. **Findings** — 1〜3件のみ。各所見に重要度ラベルを付与:
   CRITICAL = 無視すると進行を阻止または失敗を引き起こす
   HIGH     = 重大なリスクまたは逃した機会
   MEDIUM   = 検討に値するが緊急ではない
   LOW      = 軽微な改善または些細な指摘
   フォーマット: **[SEVERITY]** 所見テキスト (2-4文、具体的かつ明確に)

ルール:
- 「これは問題を引き起こすかもしれない」は無価値。「XのときYのため壊れる」が有用。
- 真に回答するためにコードを確認する必要がある場合、推測せずその旨を述べる。
- 現在のアプローチが真に優れている場合はそう述べる — その上で、引き続き監視すべき点を挙げる。
- 推論のどこかの時点で、自分の結論が間違っているためには何が真でなければならないかを
  名指しする — そしてそれが何かを変えるかどうかを判断する。
- トピックと同じ言語で記述すること。
```

**`--ctx` がアクティブな場合**、各パネリストに Read/Grep/Glob ツールを追加し、
以下のブロックを追記する (役割固有の探索フォーカスを代入する):

```
あなたはコードベースにアクセスできる。分析の前にまず探索すること。

[ROLE] の探索フォーカス:
  Critic:      テストファイル、エラーハンドリング、入力バリデーション
               問い: 「例外が握りつぶされている箇所はどこか？バリデーションされていない入力は何か？」
  Realist:     依存ファイル (package.json 等)、ビルド設定、CI設定
               問い: 「未使用の依存関係はどこか？ビルドの複雑さが潜んでいるのはどこか？」
  Architect:   データモデル、スキーマ、モジュール間依存グラフ
               問い: 「意図されたアーキテクチャを迂回しているモジュールはどこか？結合すべきでないものが結合しているのはどこか？」
  Outsider:    README、ドキュメント、公開APIサーフェス
               問い: 「ドキュメントが約束していてコードが実現していないものはどこか？」
  Contrarian:  最も古いファイル、レガシーコード、TODO/FIXMEコメント
               問い: 「古い決定が今もなお制約しているものは何か？放棄されたより良い道を示すTODOはどれか？」
```

`--ctx` を使用すると、パネリストは異なるファイルを参照する可能性がある。これは
機能である (より広いカバレッジ) がバグではない。ただし、結果を統合する際にはこの点を留意する。

### ステップ 3: 結果の提示

**サマリーを最初に** — 大半のユーザーはここしか読まない。

```markdown
---

## Discussion Panel: [トピックを約10語で]
*モード: [standard/full/max] | パネリスト: [N]名*

### サマリー
- **合意点**: [全パネリストが一致した点]
- **緊張点**: [パネリスト間で意見が分かれた点 — 双方の立場を記述。全会一致なら「なし」]
- **発見**: [元の会話では誰も提起しなかった新しい視点]

### 推論ハイライト
> [ステップ 2 の最も精緻な Reasoning chain をそのまま引用、または明瞭さのために軽く編集。パネリストを明記。]

### Findings

| # | 重要度 | 所見 | パネリスト |
|---|----------|---------|----------|
| 1 | CRITICAL | [最も深刻な所見] | [役割] |
| 2 | HIGH     | [次の所見] | [役割] |
| 3 | MEDIUM   | [所見] | [役割] |
| ... | ...   | ... | ... |

行を重要度順にソートする (CRITICAL > HIGH > MEDIUM > LOW)。
同一重要度内: Critic → Realist → Architect → Outsider → Contrarian。

### Collision Analysis
[ステップ 3.5 のサブエージェントの結果 — Standard モードではこのセクションを省略]

---

*パネル完了。どの点が響きましたか？ 深掘りしたいポイントはありますか？*
```

**発見**について: 真に新しい洞察のみを記載する。パネリストが既知の懸念事項を
繰り返しただけの場合は「なし — パネリストは新しい視点を浮上させるのではなく、
既存の懸念事項を補強した。」と記述する。水増しより誠実なフレーミングを優先する。

### ステップ 3.5: Collision Analysis

**Full モードと Max モードのみ。** Standard モード (パネリスト2名) では、
専用サブエージェントをスキップし、2つの Reasoning chain 間の緊張をインラインで
簡潔に記述する。

Full/Max の場合: **専用サブエージェント** (新鮮なコンテキスト、会話履歴なし) を起動する。
各パネリストの Reasoning Chain のみを渡す — Findings は渡さない。
これにより「答えを知った上で謎を解く」ことを防ぐ。

エージェントプロンプト:
```
あなたは Collision Analyst である。レビューパネルの Reasoning chain のみを受け取る。
結論は見ていない。

## レビュー対象のトピック
[ステップ 1 のトピック]

## Reasoning chain
[各パネリストの Reasoning Chain を役割でラベル付けして挿入 — 例: **Critic:** ...]

注: 各パネリストの推論には内部的な自己挑戦
(「自分が間違っているとしたら」という瞬間) が含まれることがある。これをパネリスト間の
矛盾として扱わないこと — これはパネリスト内の弁証法であり、パネリスト間の対立ではない。

## あなたのタスク
1. Reasoning chain 間の矛盾や緊張を特定する。
2. 各矛盾について: 「双方の推論が共に正しいとすれば、第三の結論は: [Z]」
3. 真の矛盾が存在しない場合は、正直にそう述べる。

フォーマット: **[Role A] vs [Role B]**: [矛盾]。第三の結論: [Z]。
1〜3件の衝突結果を出力する。衝突を捏造しないこと。
```

Collision Analysis の結果をステップ 3 の出力の Findings テーブルの後に含める。

### ステップ 4: ファシリテーション

結果提示後:

- パネリストの提案をすぐに採用しない
- 同様に、すぐに却下もしない
- ユーザーにどの点が響き、どの点が響かなかったかを尋ねる
- ユーザーがある点を掘り下げたい場合は、議論するか、焦点を絞ったフォローアップを起動する

パネルは情報を提供する。指示はしない。決めるのは**ユーザー**である。

### ステップ 5: 実装パイプライン (任意)

ユーザーが所見に基づいて行動したい場合 (通常、受理した所見が2件以上)、
2エージェントワークフローを提案する。注: これにより大量のトークンコストが発生する (Opus による生成 + レビュー)。

1. **Generator** (Opus — 忠実な実装には最も深い推論が必要) —
   受理した所見を番号付きチェックリスト + ファイルパスとして渡す。全て実装し、余計なものは加えない。
2. **Reviewer** — 同じチェックリスト。各項目を検証: 行参照付きで PASS/FAIL を判定。
3. レビュー結果を提示する。クローズ前に全ての FAIL を修正する。

## 使用すべきでない場合

- 明確な答えがある単純な事実の質問
- ユーザーが明確に議論ではなく実行を求めている場合
- 振り返りよりもスピードが重要な高速イテレーションループ
- 判断を誤ってもコストが無視できる些細な決定

## コスト意識

| モード | パネリスト | デフォルトモデル | おおよそのトークン倍率 |
|------|-----------|---------------|-------------------------|
| Standard | 2 (Critic, Realist) | sonnet | 約3倍 (Balanced で約4倍) |
| Full | 4 (+Architect, Outsider) | sonnet (opus 推奨) | 約5-6倍 (Balanced で約7倍) |
| Max | 5 (+Contrarian) | opus (Balanced = デフォルト) | 約7-8倍 |

`--ctx` がアクティブな場合 (技術トピックではデフォルト)、コードベース探索のため
上記の倍率に約1.5-2倍を加算する。探索が不要な技術トピックでは `--no-ctx` で抑制する。

Max + Opus は最もコストの高い構成だが、同時に最も徹底的でもある。
誤った選択のコストが分析コストを大幅に上回る場合に限り使用する。

**モデル選択の根拠 (実証テストに基づく):**
- **Haiku**: 非推奨。コードベース探索の効率が悪く、同等以下の品質で Sonnet の4倍の
  トークンを消費する。
- **Sonnet**: コスト対品質比が最良。Standard および Full モードのデフォルト。
- **Opus**: 最も深い分析。重要な判断における Full では推奨、Max では必須デフォルト。
  ユーザーはいつでも上書き可能。
